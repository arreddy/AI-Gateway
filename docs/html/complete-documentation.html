Executive Summary

What is Astra Gateway?

Astra Gateway is an enterprise-grade AI Gateway Platform - a unified control plane and proxy for all AI model traffic across organizations. It abstracts away the complexity of managing multiple LLM providers (OpenAI, Anthropic, Google, Mistral, xAI, etc.) and provides intelligent routing, governance, observability, and cost optimization.

Think of it as: The "Kong API Gateway" for AI/LLMs + "OpenRouter" routing + "Portkey" governance + enterprise operational excellence.

System Architecture

Overview

Astra Gateway is a cloud-native microservices platform built for enterprise AI workloads. It provides a unified control plane for LLM traffic management, governance, and observability.

C4 Model Architecture

Level 1: System Context

At the highest level, Astra Gateway acts as a central hub connecting multiple client applications to multiple LLM providers.

Level 2: Container Architecture

  • API Gateway (Envoy Proxy): Request validation, authentication, rate limiting, streaming coordination
  • Control Plane Services: Auth, Routing, Governance, Observability, Billing
  • Provider Adapters: Normalize requests/responses for each provider
  • Data Layer: PostgreSQL, Redis, ClickHouse, Kafka

Core Components

1. Gateway Service (Port 8080)

Purpose: Main API entry point

  • OpenAI-compatible API endpoints
  • Request validation
  • Authentication delegation
  • Load balancing across providers
  • Streaming coordination
  • Response composition

2. Auth Service (Port 9000)

Purpose: API key and JWT verification

  • API key validation (SHA-256 hashing)
  • JWT verification (RS256)
  • Permission checking
  • Claims caching (5 minute TTL)

3. Routing Engine (Port 9000)

Purpose: Intelligent provider selection

  • Cost-optimized routing
  • Latency-optimized routing
  • Quality-optimized routing
  • Fallback chain management

4. Governance Engine (Port 9000)

Purpose: Policy enforcement and security

  • PII detection and redaction
  • Prompt injection detection
  • Toxicity filtering
  • Custom policy enforcement

5. Observability Service (Port 9000)

Purpose: Metrics, logging, and tracing

  • Prometheus metrics collection
  • Event publishing (Kafka)
  • Analytics (ClickHouse)
  • Tracing (OpenTelemetry)

Request Flow

Client Request
  ↓
[API Gateway (Envoy)]
  ├─ TLS termination
  ├─ Rate limiting (DDoS)
  └─ Load balancing
  ↓
[Gateway Service - Go]
  ├─ Request validation
  ├─ Auth verification
  ├─ Cache lookup (Redis)
  ├─ Routing decision
  ├─ Governance pre-checks
  ├─ Provider call
  ├─ Governance post-checks
  └─ Metrics recording
  ↓
[Response to Client]
  ├─ Streaming (SSE/WebSocket)
  ├─ Metrics
  └─ Astra metadata

High Availability Design

Service Redundancy

  • Active-Active load balancer
  • Service Mesh (Istio) for traffic management
  • 3+ replicas minimum per microservice
  • Pod disruption budgets configured

Database High Availability

  • PostgreSQL primary with WAL and continuous archiving
  • 3 replicas (hot, warm, cold standby)
  • Automated failover with <30 second RTO
  • RPO: 0 (streaming replication)

Circuit Breaker Pattern

  • CLOSED: Healthy, pass-through
  • OPEN: Unhealthy (>10% errors), route to fallback
  • HALF-OPEN: Testing recovery with periodic requests

Scalability Architecture

Horizontal Scaling

  • API Gateway Layer: Stateless services enable unlimited horizontal scaling
  • Routing Engine: In-memory metrics cache via Redis
  • Provider Adapters: Connection pooling and configurable concurrency limits
  • Data Layer: PostgreSQL read replicas, connection pooling, sharding by tenant_id

Multi-Region Deployment

  • Global control plane with regional deployments
  • <50ms additional latency per region
  • Eventually consistent state across regions
  • Cross-region failover support

Cost Optimization Strategy

Token Cost Minimization

  1. Check cache for exact match (lexical) or similar response (semantic)
  2. Select provider based on cost constraints and quality
  3. Batch processing when applicable
  4. Execute request and record actual token usage

Database Schema

Key Tables

tenants - Multi-tenant isolation

CREATE TABLE tenants (
    id BIGSERIAL PRIMARY KEY,
    external_id UUID UNIQUE,
    name VARCHAR(255),
    tier VARCHAR(50),  -- 'free', 'starter', 'pro', 'enterprise'
    rate_limit_rpm INT,
    rate_limit_tpm INT,
    created_at TIMESTAMP
);

api_keys - Authentication

CREATE TABLE api_keys (
    id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT REFERENCES tenants,
    key_hash VARCHAR(255) UNIQUE,  -- Never plaintext
    status VARCHAR(50),
    permissions TEXT[],
    created_at TIMESTAMP,
    expires_at TIMESTAMP
);

request_logs - Immutable event log

CREATE TABLE request_logs (
    id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT REFERENCES tenants,
    api_key_id BIGINT REFERENCES api_keys,
    request_id UUID UNIQUE,
    model_requested VARCHAR(255),
    model_used VARCHAR(255),
    provider_id BIGINT,
    input_tokens INT,
    output_tokens INT,
    total_cost DECIMAL(15,8),
    latency_ms INT,
    status VARCHAR(50),
    created_at TIMESTAMP
);

Deployment Guide

Prerequisites

  • Kubernetes 1.24+
  • Helm 3.0+
  • Docker
  • Terraform (for cloud infrastructure)
  • kubectl configured

Deployment Steps

1. Prepare Infrastructure

# Create cloud infrastructure with Terraform
cd infrastructure/terraform
terraform init
terraform apply

# Output includes VPC, cluster, RDS, ElastiCache, security groups

2. Create Kubernetes Cluster

# AWS EKS
aws eks create-cluster \
  --name astra-gateway \
  --version 1.28

# GCP GKE
gcloud container clusters create astra-gateway \
  --zone us-central1-a \
  --num-nodes 3

# Azure AKS
az aks create \
  --resource-group astra-rg \
  --name astra-gateway \
  --node-count 3

3. Install Kubernetes Add-ons

# NGINX Ingress Controller
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx --create-namespace

# Cert Manager
helm repo add jetstack https://charts.jetstack.io
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager --create-namespace \
  --set installCRDs=true

# Prometheus Operator
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install prometheus prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace

4. Deploy Astra Gateway

# Create namespace
kubectl create namespace astra

# Create secrets
kubectl create secret generic astra-secrets \
  -n astra \
  --from-literal=JWT_SECRET_KEY=$(openssl rand -base64 32) \
  --from-literal=DATABASE_URL="postgresql://..." \
  --from-literal=REDIS_URL="redis://..."

# Deploy with Helm
helm repo add astra https://helm.astragateway.io
helm install astra-gateway astra/astra-gateway \
  --namespace astra \
  --values values-prod.yaml

5. Verify Deployment

# Check all pods
kubectl get pods -n astra --watch

# Test health endpoint
curl http://localhost:8080/health

# Check logs
kubectl logs -n astra -l app=gateway-service -f

Disaster Recovery

Backup Strategy

  • PostgreSQL: WAL archiving to S3 every 5 minutes, 30-day retention
  • Redis: Daily snapshot backup to S3
  • ClickHouse: Replicated tables across regions

Recovery Procedure

# 1. Restore database from backup
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier astra-gateway-restored \
  --db-snapshot-identifier arn:aws:rds:...

# 2. Update connection string
kubectl patch secret astra-secrets \
  -p '{"data":{"DATABASE_URL":"postgresql://restored-db:5432/astra"}}'

# 3. Restart services
kubectl rollout restart deployment/gateway-service -n astra

# 4. Verify health
curl http://gateway-service:8080/health

Security Architecture & Governance

Overview

Astra Gateway implements enterprise-grade security with zero-trust architecture, defense-in-depth, and compliance-ready design (SOC2, GDPR, HIPAA).

Security Layers

Layer 1: Network Security

  • TLS/SSL: TLS 1.3 for all traffic
  • DDoS Protection: AWS Shield, WAF, rate limiting
  • Network Policies: Kubernetes network policies for pod-to-pod communication
  • VPC Security: Private subnets, security groups, bastion hosts

Layer 2: Authentication & Authorization

  • API Keys: SHA-256 hashing, never plaintext storage
  • JWT Tokens: RS256 signing, expiration checks, revocation lists
  • OAuth2/OIDC: Google, GitHub, Microsoft, custom providers
  • RBAC: Role-based access control with admin, member, viewer roles
  • ABAC: Attribute-based access control for fine-grained permissions

Layer 3: Data Security

  • Encryption at Rest: AES-256 for databases and backups
  • Encryption in Transit: TLS 1.3 for all communications
  • Data Residency: Geographic enforcement of data location
  • Multi-Tenancy: Row-level security with automatic tenant_id filtering
  • PII Detection: Automatic redaction of sensitive data

Layer 4: Audit & Compliance

  • Immutable Audit Logs: Append-only event log with tamper detection
  • Compliance Tracking: SOC2, GDPR, HIPAA compliance controls
  • Access Logging: All access events recorded with timestamps and actors
  • Retention Policies: 1 year hot storage, longer archive retention

Zero Trust Architecture

Every Request Must:
  1. Verify Identity
     ├─ API Key validation
     ├─ Token signature check
     └─ Revocation list check
  
  2. Verify Device
     ├─ Client certificate verification
     ├─ Mutual TLS
     └─ Device fingerprinting (optional)
  
  3. Verify Context
     ├─ Resource access check
     ├─ Rate limit check
     └─ Quota check
  
  4. Make Decision
     └─ Allow, Deny, or Challenge

Governance Policy Engine

PII Detection & Redaction

Detects: SSN, credit cards, emails, phone numbers
Actions:
  - Redact: Replace with [REDACTED]
  - Block: Return 403 Forbidden
  - Warn: Log violation but allow
  - Alert: Notify security team

Prompt Injection Detection

  • Pattern matching for common injection attempts
  • ML-based anomaly detection
  • Quarantine suspicious requests
  • Maintain injectioncatalog for future detection

Custom Policy Engine

  • DSL for defining custom policies
  • Conditional rules based on context
  • Integration with external policy engines
  • Audit trail of policy violations

Incident Response

Detection

Response

  2. Preserve evidence (logs, metrics)

  4. Begin investigation
  5. Implement remediation
