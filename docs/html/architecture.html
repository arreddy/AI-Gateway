System Architecture

Overview

Astra Gateway is a cloud-native microservices platform built for enterprise AI workloads. It provides a unified control plane for LLM traffic management, governance, and observability.

C4 Model Architecture

Level 1: System Context

At the highest level, Astra Gateway acts as a central hub connecting multiple client applications to multiple LLM providers:

  • Clients: Mobile apps, web apps, backend services
  • Gateway: Unified API endpoint with OpenAI compatibility
  • Providers: OpenAI, Anthropic, Google Gemini, and 50+ others

Level 2: Container Architecture

The platform is organized into logical containers within the Astra Gateway system:

  • API Gateway (Envoy Proxy): Request validation, authentication, rate limiting, streaming coordination, response caching
  • Control Plane Services: Auth, Routing, Governance, Observability, Billing
  • Provider Adapters: Normalize requests/responses for each provider
  • Data Layer: PostgreSQL, Redis, ClickHouse, Kafka

Level 3: Component Breakdown

API Gateway Service

The core request processing pipeline flows through the gateway service with the following stages:

Client Request → Envoy Proxy → Gateway Service → Routing Engine 
→ Provider Adapter → External Provider → Response Processing → Client

Authentication Flow

API key validation happens at the API Gateway layer with caching:

  1. Extract API key from Authorization header
  2. Quick cache lookup (TTL: 5 min)
  3. If cache miss, call Auth Service via gRPC
  4. Verify signature, check expiration, validate permissions
  5. Cache result and return claims object

Routing Decision Engine

Intelligent provider selection based on multiple factors:

  • Load Metrics: Provider latencies, error rates, costs, availability
  • Routing Policies: Cost-optimized, latency-optimized, quality-optimized, adaptive
  • Constraints: Budget, rate limits, geographic restrictions, data residency
  • Fallback Chain: Primary, secondary, and tertiary providers

Governance Pipeline

Policy enforcement at multiple stages:

  1. PII Detector - Scan for SSN, credit cards, emails
  2. Prompt Injection Detector - Pattern matching and ML-based detection
  3. Toxicity Filter - Content classification and policy enforcement
  4. Custom Policy Engine - DSL evaluation and conditional rules
  5. Audit Logger - Immutable event recording

Multi-Tenant Request Isolation

Complete tenant isolation is enforced at every level:

  • All queries filtered by tenant_id automatically
  • Separate rate limit buckets per tenant
  • Isolated cache keys (includes tenant_id)
  • Isolated quota tracking
  • Cross-tenant data leakage: IMPOSSIBLE

Scalability Architecture

Horizontal Scaling

  • API Gateway Layer: Stateless services enable unlimited horizontal scaling
  • Routing Engine: In-memory metrics cache via Redis
  • Provider Adapters: Connection pooling and configurable concurrency limits
  • Data Layer: PostgreSQL read replicas, connection pooling, sharding by tenant_id

Multi-Region Deployment

Distributed architecture with eventual consistency:

  • Global Control Plane: Single source of truth for config and policies
  • Regional Deployments: Gateway, cache, database per region
  • Cross-Region Sync: Eventually consistent state
  • Geo-Routing: <50ms additional latency per region

High Availability Design

Service Redundancy

  • Active-Active load balancer
  • Service Mesh (Istio) for traffic management
  • 3+ replicas minimum per microservice
  • Pod disruption budgets configured

Database High Availability

  • PostgreSQL primary with WAL and continuous archiving
  • 3 replicas (hot, warm, cold standby)
  • Automated failover with <30 second RTO
  • RPO: 0 (streaming replication)

Circuit Breaker Pattern

Automatic provider health tracking prevents cascade failures:

  • CLOSED: Healthy, pass-through
  • OPEN: Unhealthy (>10% errors), route to fallback
  • HALF-OPEN: Testing recovery with periodic requests

Security Architecture

Defense in Depth

  • Layer 1 - Network: WAF, DDoS protection, VPC security, TLS 1.3
  • Layer 2 - Application: API key validation, JWT verification, input validation, rate limiting
  • Layer 3 - Data: Encryption at rest (AES-256), encryption in transit (TLS 1.3), PII detection, tenant isolation
  • Layer 4 - Monitoring: Immutable audit logs, security event monitoring, anomaly detection, compliance tracking

Zero Trust Architecture

Every request is verified at multiple levels:

  1. Verify identity (API Key, JWT, OAuth2)
  2. Verify device (mTLS certificates)
  3. Verify context (resource, action, data sensitivity)
  4. Make decision (allow, deny, challenge)

Cost Optimization Strategy

Token Cost Minimization

  1. Check cache for exact match (lexical cache) or similar response (semantic cache)
  2. Select provider based on cost constraints and quality requirements
  3. Batch processing when applicable
  4. Execute request and record actual token usage
