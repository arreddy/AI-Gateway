Security Architecture & Governance

Overview

Astra Gateway implements enterprise-grade security with zero-trust architecture, defense-in-depth, and compliance-ready design (SOC2, GDPR, HIPAA).

Network Security

TLS/SSL Encryption

Client ←──TLS 1.3──→ Load Balancer
                      ↓
                  Gateway Service
                      ↓
                  Internal Services (mTLS)

Configuration

tls:
  min_version: "1.3"
  cipher_suites:
    - TLS_AES_256_GCM_SHA384
    - TLS_CHACHA20_POLY1305_SHA256
    - TLS_AES_128_GCM_SHA256
  certificate_provider: "certmanager"

DDoS Protection

  • AWS Shield (managed DDoS protection)
  • Rate limiting at edge (WAF)
  • Geo-blocking if needed

Network Policies

# Only allow traffic from ingress controller to gateway
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: gateway-ingress-only
spec:
  podSelector:
    matchLabels:
      app: gateway-service
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app.kubernetes.io/name: ingress-nginx

Authentication & Authorization

API Key Security

# Key format: sk_prod_[32 random bytes]
# Never stored in plaintext
# Only SHA-256 hash in database
# Key rotation policy:
#   - Issued keys expire after 1 year
#   - Old key remains valid 30 days during transition

JWT Tokens

{
  "sub": "user_456",
  "org": "org_123",
  "iss": "astra-gateway",
  "aud": "astra-api",
  "iat": 1705334400,
  "exp": 1705420800,
  "permissions": ["completions:create", "models:list"],
  "rate_limits": { "rpm": 60000 }
}

OAuth2/OIDC

Supported providers:

  • Google (oauth2/google)
  • GitHub (oauth2/github)
  • Microsoft (oauth2/microsoft)
  • Custom OIDC provider

Data Security

Encryption at Rest

  • PostgreSQL: Transparent Data Encryption (TDE), master key in AWS KMS, quarterly rotation
  • Redis: Optional encryption for sensitive data, TTL for cache entries
  • ClickHouse: Replication to encrypted replicas, backups encrypted

Data Residency & Compliance

# Data residency controls
residency_controls:
  us_region:
    allowed_providers: ["us-east-1", "us-west-2"]
    prohibited_providers: ["eu-west-1"]
  
  eu_region:
    allowed_providers: ["eu-west-1", "eu-central-1"]
    prohibited_providers: ["us-east-1"]

Multi-Tenancy Isolation

Row-level security enforces tenant isolation:

  • Every query automatically filtered by tenant_id
  • Tenant A cannot see Tenant B's data
  • Cross-tenant joins impossible
  • Backups include only single tenant

Input Validation & Protection

Request Validation

  1. Schema validation: Check required fields, validate types, verify ranges, limit request size (50MB)
  2. Content validation: JSON format, message length, model names, parameter ranges
  3. Business logic validation: API key active, quota available, provider configured

Injection Detection

  • Prompt Injection: Pattern matching + ML model detection
  • Command Injection: Safe APIs only, no shell execution
  • SQL Injection: Parameterized queries only, ORM protection
  • XXE Prevention: XML parsing disabled by default

Audit & Compliance

Immutable Audit Logs

CREATE TABLE audit_logs (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    action VARCHAR(50),       -- 'create', 'read', 'update', 'delete'
    resource_type VARCHAR(50), -- 'api_key', 'policy', 'request'
    actor_type VARCHAR(50),    -- 'user', 'system', 'api'
    changes JSONB,             -- Before/after values
    ip_address INET,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Prevent deletion
CREATE TRIGGER prevent_audit_log_delete
    BEFORE DELETE ON audit_logs
    FOR EACH ROW
    EXECUTE FUNCTION raise_exception();

Compliance Frameworks

  • SOC2: Logical access controls, system monitoring, incident reporting
  • GDPR: Data processing agreements, data residency, right to be forgotten, breach notification
  • HIPAA: PHI audit controls, encryption, breach notification procedures
  • ISO 27001: Information security management system

policy := GovernancePolicy{

Zero Trust Architecture

Every Request
  ├─ 1. Verify Identity
  │  ├─ API Key validation
  │  ├─ Token signature check
  │  └─ Revocation list check
  │
  ├─ 2. Verify Device
  │  ├─ Client certificate verification
  │  ├─ Mutual TLS
  │  └─ Device fingerprinting (optional)
  │
  ├─ 3. Verify Network
  │  ├─ IP whitelisting
  │  ├─ Geo-location check
  │  └─ VPN/Proxy detection
  │
  ├─ 4. Verify Resource
  │  ├─ Tenant isolation check
  │  ├─ API key permission check
  │  └─ Rate limit check
  │
  └─ Decision: Allow, Deny, or Challenge

Incident Response

Detection

  • Automated alerting on security events
  • Real-time monitoring of audit logs
  • Anomaly detection ML models

Response Procedures

  1. Isolate affected service
  2. Preserve evidence (logs, metrics)
  3. Notify security team
  4. Begin investigation
  5. Communicate with affected customers

Recovery

  1. Patch or rollback
  2. Verify remediation
  3. Resume normal operations
  4. Post-incident review

Security Checklist

Pre-Deployment Verification:
  • ☐ TLS 1.3 enabled for all traffic
  • ☐ API keys hashed with SHA-256
  • ☐ JWTs signed with RS256
  • ☐ All database queries parameterized
  • ☐ Row-level security enforced
  • ☐ Audit logs immutable and encrypted
  • ☐ Rate limiting active
  • ☐ DDoS protection enabled
  • ☐ Secrets encrypted at rest
  • ☐ Regular penetration testing
  • ☐ Incident response plan in place
  • ☐ SOC2 audit scheduled
  • ☐ GDPR compliance verified
  • ☐ Data residency policies enforced
